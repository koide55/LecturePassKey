# 第1回 パスキー入門

## この教材の目的

この講義ノートは、パスキーの技術を初学者から実装者の入口まで橋渡しするための導入資料です。単に「便利なログイン手段」として理解するのではなく、従来のパスワード認証が抱えていた問題、FIDO2 と WebAuthn が何を解決しようとしているのか、実装時にどこで注意が必要かを一段ずつ整理します。

このリポジトリでは、次の 2 本立てで学べるようにします。

- `slides/`: 概念理解のための解説資料
- `app/`: 実際に登録、認証、失敗パターンを試すハンズオン用アプリ

## 1. なぜパスキーが必要なのか

### パスワード認証の限界

長年の Web サービスでは、利用者が ID とパスワードを入力し、サーバがその照合結果でログインを許可する方式が中心でした。しかしこの方式には、構造的な弱点があります。

- 利用者が弱いパスワードを選びやすい
- 使い回しが起こりやすい
- フィッシングサイトに入力してしまうと秘密がそのまま盗まれる
- サーバ側でハッシュ化していても漏えい時の影響が大きい
- MFA を追加しても、運用が複雑になったり中間者攻撃に弱い場合がある

要するに、パスワードは「覚えて再入力する秘密」です。この性質自体が、盗難と再利用に弱い理由です。

### パスキーの狙い

パスキーは、利用者が秘密文字列を覚えて入力する代わりに、端末が保持する公開鍵暗号の資格情報を使って本人確認する仕組みです。大きな狙いは次のとおりです。

- 秘密をサーバに送らない
- 秘密そのものを人間に入力させない
- 正しいオリジンでしか使えないようにする
- 生体認証や端末ロック解除と組み合わせて使いやすくする

## 2. パスキーとは何か

### ざっくりした定義

パスキーは、一般には FIDO2 / WebAuthn ベースの公開鍵資格情報を、OS やブラウザが使いやすい形で提供する仕組みを指します。

利用者から見える体験は次のようになります。

1. サイトで「パスキーを登録」を選ぶ
2. 端末の Face ID、Touch ID、Windows Hello、画面ロックなどで本人確認する
3. 以後はパスワード入力の代わりに、その端末からサインインできる

### 本質を一言でいうと

パスキーの本質は、「秘密の文字列をサーバと共有する」のをやめて、「端末の中にある秘密鍵で署名し、サーバは公開鍵でそれを確かめる」ことです。

パスワードの世界では、利用者とサーバが同じ秘密に依存しています。だから、その秘密が盗まれると第三者も本人になれてしまいます。

パスキーの世界では、次のように役割が分かれます。

- 端末:
  - 外に出さない秘密鍵を持つ
- サーバ:
  - 秘密鍵に対応する公開鍵だけを持つ

つまり、サーバは本人の秘密そのものを預からずに、本人らしさを確認できます。ここがいちばん大きな転換です。

### 公開鍵の仕組みをシンプルに理解する

公開鍵暗号では、鍵が 2 つあります。

- 秘密鍵
  - 本人の端末だけが持つ
  - 外に出さない
- 公開鍵
  - サーバに渡してよい
  - 署名が本物か確かめるために使う

イメージとしては、秘密鍵が「本人しか押せない印鑑」、公開鍵が「その印鑑が本物か見分けるための照合情報」です。

サーバはログインのたびに challenge という毎回変わるお題を出します。端末はそのお題に対して秘密鍵で署名します。サーバは公開鍵を使って、その署名が本物で、しかも今出したお題に対するものかを確認します。

この方式だと、認証のたびに毎回同じ秘密を送る必要がありません。だから、パスワードのように「入力したものを盗まれて再利用される」危険を大きく減らせます。

### 技術的な正体

パスキーの中心にあるのは公開鍵暗号です。

- 端末側:
  - 秘密鍵を安全に保持する
- サーバ側:
  - 公開鍵を保存する
- 認証時:
  - サーバが challenge を出す
  - 端末が秘密鍵で署名する
  - サーバが公開鍵で検証する

このため、サーバが持つのは公開鍵であり、パスワードのような「漏れたら終わりの共有秘密」ではありません。

### パスワードとの違いをもっと直感的に見る

- パスワード:
  - 本人が秘密を知っていることを示す
  - その秘密を入力する
  - 盗まれると他人も同じ秘密を使える
- パスキー:
  - 本人の端末が秘密鍵を持っていることを使う
  - 秘密鍵そのものは送らない
  - 署名結果だけを送る

ここで重要なのは、「秘密を送らない」のに「本人の端末が正しい秘密を持っている」と証明できることです。これが公開鍵方式の強みです。

### まず覚えてほしい最小ポイント

パスキーを学び始める段階では、まず次の 3 点を押さえると全体が見えやすくなります。

1. パスキーはパスワードを別の UI にしたものではない
2. 端末内の秘密鍵とサーバ側の公開鍵で認証している
3. 毎回 challenge に署名するので、使い回しやフィッシングに強い

この 3 つが分かると、以後の WebAuthn の登録処理や認証処理も理解しやすくなります。

### 利用者から見たパスキーの体験

パスキーは、利用者にとって「難しい暗号技術を使う仕組み」である前に、「毎回パスワードを思い出して入力しなくてよいログイン体験」でもあります。

多くの環境では、利用者は次のような方法で本人確認します。

- 指紋認証
- 顔認証
- 端末の PIN
- 画面ロック解除

ここで大事なのは、生体認証や PIN そのものが Web サイトに送られるわけではないことです。これらは、端末の中にある秘密鍵を使ってよい本人かどうかを、端末側で確かめるために使われます。

つまり、利用者の感覚としては「指紋や顔でログインしている」ように見えますが、技術的には「端末が秘密鍵を使って署名している」構造です。生体情報や PIN をサイトに送って照合しているわけではありません。

### パスワード関連情報を外に出さない

パスワード認証では、利用者は毎回パスワードを入力し、その内容に依存してサーバ側が認証します。そのため、入力先が偽サイトだったり、サーバ側で漏えいが起きたりすると、パスワード関連情報が悪用される余地があります。

パスキーでは、そもそも「サイトに送って照合してもらう秘密の文字列」がありません。

- 端末の外に出さない秘密鍵を使う
- サーバには公開鍵だけを登録する
- ログイン時には署名結果を返す

このため、パスワードそのものや、パスワード由来の共有秘密を外部に出さずに認証できます。利用者は秘密を打ち込んで渡す必要がなく、サーバもその秘密を保管しません。

### パスワードとの違いを表で見る

| 観点 | パスワード | パスキー |
|---|---|---|
| 利用者の操作 | 文字列を思い出して入力する | 生体認証や端末 PIN で確認する |
| サイトに渡すもの | パスワード、またはその検証に必要な情報に依存する | challenge に対する署名結果 |
| サーバが主に保持するもの | パスワードハッシュ | 公開鍵、credential ID |
| 端末の中に残る秘密 | 必ずしも安全に保護されないことがある | 秘密鍵が端末内に保持される |
| フィッシングへの強さ | 偽画面に入力すると盗まれやすい | オリジンに結び付くため盗まれにくい |
| サーバ漏えい時の影響 | ハッシュ悪用や使い回し被害につながりやすい | 公開鍵だけではそのままログインしにくい |
| 使い勝手 | 記憶、入力、再設定が必要 | 端末の普段の解除操作に近い |

この表のポイントは、パスキーが単に「パスワードを別の見た目にしたもの」ではなく、利用者体験と認証の仕組みの両方を変えていることです。

## 3. 関連用語の整理

### FIDO2

FIDO2 は、パスワードレス認証やフィッシング耐性のある認証を実現するための枠組みです。実務上は、主に次の 2 要素を合わせたものとして理解すると分かりやすいです。

- WebAuthn
  - ブラウザと Web アプリのための標準 API
- CTAP
  - 端末やセキュリティキーとクライアントのやり取りを定めるプロトコル

### WebAuthn と FIDO2 の技術的な位置づけ

この 2 つは似た言葉に見えますが、同じものではありません。

- FIDO2:
  - パスワードレス認証を実現するための全体的な枠組み
- WebAuthn:
  - その中で Web ブラウザと Web アプリが使う標準 API

つまり、Web サービス開発者の目線では、ブラウザから触る入口が WebAuthn であり、その背後にある大きなエコシステムや考え方を含むのが FIDO2 です。

もう少し分けて言うと、役割は次のようになります。

- WebAuthn:
  - Web ページの JavaScript から認証器を使うための仕様
- CTAP:
  - ブラウザや OS と、認証器やセキュリティキーが通信するための仕様
- FIDO2:
  - それらを組み合わせた認証の枠組み

### ざっくりした関係図

1. Web アプリはブラウザで `navigator.credentials.create()` や `get()` を呼ぶ
2. これは WebAuthn の API として提供される
3. ブラウザや OS は必要に応じて認証器と連携する
4. その連携部分で CTAP が使われる
5. この全体像を含めて、一般に FIDO2 系の認証と呼ぶ

つまり、WebAuthn は「Web アプリから見える窓口」、FIDO2 は「その窓口の背後も含めた全体像」と考えると整理しやすくなります。

### WebAuthn

WebAuthn は W3C の標準で、JavaScript から認証器を使うための API です。フロントエンド実装では主に次の 2 つを使います。

- `navigator.credentials.create()`
  - 新しい資格情報を登録する
- `navigator.credentials.get()`
  - 既存の資格情報で認証する

### 認証器

認証器は秘密鍵を使って署名を行う存在です。たとえば次のようなものがあります。

- スマートフォン内蔵の認証器
- ノート PC の生体認証基盤
- USB セキュリティキー

## 4. 登録フロー

パスキー登録時の流れを、概念レベルで追います。

1. 利用者がログイン済み状態で「パスキー登録」を押す
2. サーバがランダムな challenge を生成する
3. サーバが RP 情報、ユーザ情報、challenge などをブラウザへ返す
4. ブラウザが `navigator.credentials.create()` を呼ぶ
5. 端末が本人確認を行い、鍵ペアを生成する
6. 公開鍵付きの attestation あるいは登録結果がサーバに返る
7. サーバが challenge などを検証し、公開鍵と credential ID を保存する

### 登録でサーバが保存するもの

実装によって差はありますが、基本的には以下のような情報を保存します。

- ユーザ ID
- credential ID
- 公開鍵
- sign count
- transports
- 作成日時やラベル

秘密鍵はサーバに保存しません。

---

## 5. 認証フロー

認証時の流れも同様に見ていきます。

1. 利用者が「パスキーでログイン」を選ぶ
2. サーバが challenge と許可 credential 候補を返す
3. ブラウザが `navigator.credentials.get()` を呼ぶ
4. 認証器が challenge とオリジンに基づき署名する
5. ブラウザが assertion をサーバへ送る
6. サーバが署名、challenge、origin、RP ID を検証する
7. 検証に成功したらログイン成功とする

### パスワード認証と何が違うか

- パスワード:
  - 利用者が秘密を入力する
  - サーバ側は共有秘密の検証をする
- パスキー:
  - 利用者は秘密を入力しない
  - 端末が秘密鍵で署名する
  - サーバ側は公開鍵で署名検証をする

## 6. フィッシング耐性が高い理由

パスキーが注目される最大の理由の一つは、フィッシング耐性です。

### なぜ強いのか

- 認証はオリジンに束縛される
- 偽サイトでは正しい RP ID / origin の条件を満たせない
- パスワードのように「入力内容を盗んで再利用」がしにくい

たとえば、`example.com` 用に登録された資格情報は、見た目がそっくりな `examp1e.com` ではそのまま使えません。ここがパスワードとの大きな違いです。

### フィッシング耐性をもっとシンプルにいうと

パスワードは、利用者が見ている画面にそのまま打ち込む方式です。だから、偽サイトでも利用者が信じて入力してしまえば盗まれます。

一方でパスキーは、利用者が文字列を相手に渡すのではなく、端末が「このサイトのための秘密鍵」を使って署名します。そのため、偽サイトは本物サイト向けの資格情報をそのまま引き出しにくくなります。

要するに、パスワードは「人が秘密を渡す」方式で、パスキーは「端末がそのサイト向けの署名を返す」方式です。ここがフィッシング耐性の源です。

## 7. なぜ安全なのか

パスキーが安全だと言われるのは、単に新しいからではありません。攻撃者にとって奪いやすかったものが、仕組みの段階で奪いにくくなっているからです。

ここでは、特に重要な 3 つの観点を見ます。

- フィッシング耐性
- サーバ漏えい耐性
- リプレイ攻撃耐性

### 7-1. フィッシング耐性

フィッシング耐性とは、偽サイトにだまされても認証情報をそのまま奪われにくい性質です。

パスキーでは、認証器がオリジンや RP ID と結び付いた条件で動きます。つまり、`example.com` 向けに作られた資格情報は、別ドメインの偽サイトではそのまま正しく使えません。

なぜなら、サーバが検証しているのは単なる「文字列が一致したか」ではなく、次のような条件だからです。

- 今回の challenge に対する署名か
- 想定した origin / RP ID に結び付く署名か
- 登録済み credential に対応する署名か

このため、攻撃者が見た目だけ本物そっくりなログイン画面を作っても、パスワードのように認証情報をその場で抜き取って使い回すのが難しくなります。

### 7-2. サーバ漏えい耐性

サーバ漏えい耐性とは、サーバ側のデータベースが漏れても、そこからすぐ本人としてログインされにくい性質です。

パスワード認証では、サーバはパスワードハッシュを持っています。ハッシュ化されていても、漏えい後に総当たりや使い回し悪用の対象になります。

一方でパスキーでは、サーバが保存しているのは主に次の情報です。

- 公開鍵
- credential ID
- 利用メタデータ

ここで重要なのは、公開鍵だけでは署名を作れないことです。ログインに必要なのは秘密鍵であり、それは利用者端末の中にあります。

つまり、サーバが侵害されても、攻撃者が手に入れるのは「本人確認に使う照合情報」であって、「本人になりすますための秘密そのもの」ではありません。これが、パスワード方式に対する大きな強みです。

もちろん、サーバ侵害で被害がゼロになるわけではありません。ユーザ情報の漏えいや、登録済み credential の列挙、セッション情報の漏えいなどは別途問題になります。ただし少なくとも、「漏れた認証情報をそのまま使って即ログイン」という事態は起こりにくくなります。

### 7-3. リプレイ攻撃耐性

リプレイ攻撃とは、過去に盗み見た正しい認証データをもう一度送り直して、本人になりすまそうとする攻撃です。

パスキーは、この攻撃にも強くなるように設計されています。理由は、認証が毎回同じ内容ではなく、サーバが毎回新しい challenge を出すからです。

流れを単純化すると次のようになります。

1. サーバが今回限りの challenge を出す
2. 端末がその challenge に対して署名する
3. サーバが「今回出した challenge と一致するか」を確かめる

このため、以前の通信で得た署名データをそのまま再送しても、challenge が違えば通りません。

パスワード方式だと、攻撃者が正しい秘密を知ってしまえば何度でも再利用できます。しかしパスキーでは、毎回の challenge にひも付いた署名が必要なので、過去の認証結果のコピペでは突破しにくくなります。

### まず押さえるべき安全性のまとめ

- フィッシング耐性:
  - 偽サイトで認証情報をそのまま奪いにくい
- サーバ漏えい耐性:
  - サーバに秘密鍵がないので、漏れても即なりすましに直結しにくい
- リプレイ攻撃耐性:
  - 毎回異なる challenge に署名するので、過去の認証データを使い回しにくい

この 3 点が、パスキーが「便利だから」だけでなく、「構造的に安全性を上げやすい」と言われる理由です。

### それでも残る注意点

ただし「すべての攻撃が消える」わけではありません。

- セッション乗っ取り
- 登録済み端末を奪われた場合のリスク
- 回復フローの弱さ
- アカウント復旧時の本人確認不備
- サーバ実装ミスによる challenge 検証漏れ

パスキーは万能鍵ではなく、認証の土台を強化する技術です。

## 8. 実装の要点

パスキー対応アプリを作るとき、実装者はクライアントとサーバの両方を理解する必要があります。

### クライアント側の要点

- `PublicKeyCredentialCreationOptions` を正しく受け取る
- Base64URL と ArrayBuffer の変換を扱う
- `navigator.credentials.create()` / `get()` の例外を処理する
- ユーザがキャンセルした場合の導線を用意する

### サーバ側の要点

- challenge を安全に生成し、短時間だけ保持する
- 返ってきたレスポンスの challenge を必ず照合する
- origin と RP ID を検証する
- 署名検証を正しく行う
- credential ごとの sign count やメタデータを管理する

### よくある難所

- バイナリデータのエンコード変換
- ブラウザごとの差異
- 開発環境での HTTPS 要件
- 複数端末、複数 credential の管理

## 9. 学習のための脅威モデル

この教材では、単に「動いた」で終わらせず、どこを壊すと危険かも確認できるようにします。ハンズオンでは次の観点を扱う予定です。

- challenge を固定値にしたら何が起きるか
- origin 検証を外したら何が危険か
- ログイン後セッションの扱いを誤ると何が起きるか
- パスキー登録済みでもアカウント復旧が弱いとどうなるか
- UI が分かりにくいと利用者がどこで詰まるか

つまり、パスキーは「公開鍵を使えば終わり」ではなく、周辺設計まで含めて学ぶ必要があります。

## 10. ハンズオンアプリの構想

`app/` には、学習向けに次のような機能を持つ小さな Web アプリを作る予定です。

- ユーザ登録
- パスワードログイン
- パスキー登録
- パスキー認証
- 登録 credential 一覧表示
- 安全版と不安全版の比較モード

### 学習体験のねらい

- まずは正常系で登録と認証を体験する

## 11. 今回のまとめ

- パスキーは、公開鍵暗号を使ったフィッシング耐性の高い認証方式
- 利用者は秘密文字列を覚えて入力しなくてよい
- サーバは公開鍵を保存し、署名を検証する
- WebAuthn の登録と認証は challenge ベースで進む
- 実装では origin、RP ID、challenge の検証が非常に重要
- 学ぶときは正常系だけでなく、失敗系と脅威モデルも見るべき
